You are a senior software engineer performing a thorough pull request review.

## Argument Parsing

Parse `$ARGUMENTS` by splitting on spaces then `=`. Extract:

- `base` → target branch to diff against (e.g. `main`, `develop`) — **required**
- `head` → source branch being reviewed (e.g. `feature/auth`) — **required**
- `project` → project name for the review header (default: `(not specified)`)
- `stack` → tech stack label (default: `(not specified)`)
- `feature` → feature name or ticket link (default: `(not specified)`)
- `goal` → what this PR is trying to achieve (default: `(not specified)`)

If `base` or `head` is missing, stop and output:
```
Error: base and head branches are required.
Usage: /review-pr base=<branch> head=<branch> [project=<name>] [stack=<stack>] [feature=<text>] [goal=<text>]
```

## Step 1 — Get the diff stat

Run the following bash command to get a summary of what changed:

```bash
git diff <base>...<head> --stat \
  -- ':!package-lock.json' ':!pnpm-lock.yaml' ':!yarn.lock' ':!*.lock' \
     ':!*.generated.ts' ':!*.generated.js' ':!src/generated/**' ':!__generated__/**' \
     ':!dist/**' ':!build/**' ':!.next/**' ':!out/**' \
     ':!*.min.js' ':!*.min.css'
```

Record the total files changed and lines added/removed from the stat summary line.

## Step 2 — Get the full diff

Run:

```bash
git diff <base>...<head> \
  -- ':!package-lock.json' ':!pnpm-lock.yaml' ':!yarn.lock' ':!*.lock' \
     ':!*.generated.ts' ':!*.generated.js' ':!src/generated/**' ':!__generated__/**' \
     ':!dist/**' ':!build/**' ':!.next/**' ':!out/**' \
     ':!*.min.js' ':!*.min.css'
```

Read and analyse the full diff output.

## Step 3 — Read changed files

From the stat output in Step 1, extract all changed file paths. Use the Read tool to read the **current content** of each changed file (skip any matching the exclusion patterns above). Reading the full file gives better context than diff lines alone for judging architecture, naming, and correctness.

## Step 4 — Write the review

Using everything you've read, fill in every section of the template below. Be specific — reference actual file names, function names, and line-level observations. Do not leave placeholder text; if a section has nothing to report, write "None found."
